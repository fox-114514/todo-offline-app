# ToDo 任务管理 App — 需求文档

> 交给 AI agent 从零开始开发。技术栈自选，但要满足以下需求。

---

## 一、产品定位

一个**个人任务管理**应用，有后端服务 + 安卓客户端。

核心特点：
- 简洁好用，不臃肿
- **离线优先**——没网也能完整使用
- 服务端可能不稳定，客户端不能依赖服务器在线

---

## 二、功能需求

### 2.1 用户系统

- 注册（用户名 + 邮箱 + 密码）
- 登录（返回 token 用于鉴权）
- 登出（token 失效）
- 已登录过的用户在离线时允许继续进入 App，使用本地数据；网络恢复后再校验 token 或重新登录

### 2.2 任务管理

**任务字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| ID | UUID | 客户端生成，离线创建时也可立即使用 |
| 标题 | 必填，1-200字 | 任务名称 |
| 内容 | 选填，最长5000字 | 详细描述 |
| 状态 | 枚举 | 想做 / 进行中 / 完成 / 放弃 |
| 分类 | 枚举 | 游戏 / 程序 / 技能 / 其他 |
| 优先级 | 枚举 | 高 / 中 / 低 |
| 创建时间 | 自动 | |
| 更新时间 | 自动 | |
| 版本号 | 整数，自动递增 | 用于并发冲突检测 |
| 删除时间 | 可空 | 软删除标记，未删除时为空 |

**操作：**

- 创建任务
- 查看任务列表（支持按状态/分类/优先级筛选，支持搜索标题和内容，支持排序，支持分页）
- 查看任务详情
- 编辑任务（需传版本号，版本不匹配时提示冲突）
- 删除任务（软删除，不真正丢失数据）
- 随机抽取一个未完成的任务（不知道做什么的时候用）

### 2.3 提醒功能

- 每个用户有一套提醒设置：开启/关闭、提醒频率
- 提醒频率服务端和本地统一用 `frequencySeconds` 存储
- 频率范围：300 秒（5 分钟）~ 86400 秒（24 小时）
- 客户端根据设置本地弹通知

## 三、离线优先（核心要求）

> 这是本项目最重要的技术要求。服务端不稳定，客户端必须独立可用。

### 设计原则

1. **所有数据在本地有完整副本**
2. **读操作只读本地**，不等网络
3. **写操作先写本地**，再异步同步到服务器
4. **网络恢复时自动同步**
5. **冲突有明确的解决策略**

### 同步机制

- 客户端维护一个待同步队列
- 每次本地修改（增/改/删）加入队列
- 网络可用时，按顺序把队列里的操作推送到服务器
- 同步成功后标记为已同步
- 同步失败（网络断开/服务器挂了）保持待同步状态，下次重试
- 队列里的每条操作至少包含：`operationId`、`taskId`、`operationType`、`payload`、`baseVersion`、`localUpdatedAt`、`retryCount`
- `operationId` 使用 UUID，服务端需要支持幂等处理，避免重复提交造成重复创建或重复更新
- 删除任务也是同步操作，使用软删除墓碑（`deletedAt`），不能因为下次拉取又恢复出来

### 冲突处理

默认使用**同字段最后写入胜出（Last-Write-Wins）**策略：

- 客户端更新时携带 `baseVersion`
- 服务端发现版本不匹配时返回 409，并带上服务器当前任务数据
- 客户端比较本地修改和服务器数据
- 如果双方修改的是不同字段，自动合并
- 如果双方修改了同一个字段，比较本地修改时间和服务器更新时间，时间更新的一方胜出
- 如果自动合并会丢失用户输入，生成冲突提示，让用户选择保留本地版本、保留服务器版本或手动合并

### 首次同步

- 登录后，从服务器拉取全量数据到本地
- 之后的同步只处理增量变更

### 增量同步

- 服务端为每个用户维护同步游标 `syncCursor`
- 客户端本地保存最近一次成功同步的 `syncCursor`
- 拉取增量时只返回该游标之后变化的数据，包括新增、更新和软删除记录
- 推送本地队列成功后，客户端再拉取一次增量，确保本地状态和服务端最终一致
- 如果游标失效或服务端检测到数据不完整，客户端重新执行全量同步

## 四、API 设计

后端提供 RESTful JSON API，统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

JSON 字段使用 camelCase。

### 端点清单

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| POST | /api/auth/register | 否 | 注册 |
| POST | /api/auth/login | 否 | 登录 |
| POST | /api/auth/logout | 是 | 登出 |
| GET | /api/tasks | 是 | 任务列表（筛选+分页） |
| GET | /api/tasks/random | 是 | 随机任务 |
| GET | /api/tasks/{id} | 是 | 任务详情 |
| POST | /api/tasks | 是 | 创建任务 |
| PUT | /api/tasks/{id} | 是 | 更新任务（需传 version） |
| DELETE | /api/tasks/{id} | 是 | 删除任务 |
| GET | /api/reminders | 是 | 获取提醒设置 |
| PUT | /api/reminders | 是 | 更新提醒设置 |
| GET | /api/sync/pull | 是 | 拉取全量或增量变更 |
| POST | /api/sync/push | 是 | 推送客户端待同步操作 |

**分页参数**：`page`（默认1）、`pageSize`（默认20）

**筛选参数**：`status`、`category`、`priority`、`search`、`sortBy`、`sortOrder`

**认证方式**：`Authorization: Bearer <token>`，token 有效期 1 小时

**更新冲突**：PUT 时 version 不匹配返回 409

**软删除**：DELETE 后数据仍存在，查询时自动过滤

### 同步 API 说明

#### GET /api/sync/pull

用于首次同步和增量同步。

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| cursor | string | 否 | 上次同步成功后保存的游标；为空时返回全量数据 |

**响应数据：**

```json
{
  "tasks": [],
  "reminderSettings": {},
  "nextCursor": "cursor_value",
  "isFullSync": false
}
```

#### POST /api/sync/push

用于把客户端待同步队列推送到服务端。服务端需要根据 `operationId` 做幂等处理。

**请求数据：**

```json
{
  "operations": [
    {
      "operationId": "uuid",
      "taskId": "uuid",
      "operationType": "create",
      "baseVersion": 1,
      "localUpdatedAt": "2026-06-08T12:00:00Z",
      "payload": {}
    }
  ]
}
```

**响应数据：**

```json
{
  "accepted": ["operation_uuid"],
  "conflicts": [
    {
      "operationId": "operation_uuid",
      "taskId": "task_uuid",
      "serverTask": {}
    }
  ],
  "nextCursor": "cursor_value"
}
```

## 五、安卓客户端 UI

### 页面

1. **登录/注册页** — 用户名 + 密码（注册额外要邮箱）
2. **任务列表页** — 主页面，展示任务卡片，支持筛选/搜索，右下角 FAB 新建
3. **任务详情/编辑页** — 查看详情，点击编辑，修改状态/分类/优先级
4. **设置页** — 提醒开关/频率、账号信息、同步状态显示

### UI 要求

- Material Design 3 风格
- 任务卡片显示：标题、状态标签、分类标签、优先级颜色、更新时间
- 状态用颜色区分（如：进行中=蓝色，完成=绿色，放弃=灰色）
- 同步状态指示（如顶栏小图标：已同步✓ / 同步中⟳ / 待同步!）
- 离线时顶部显示 "离线模式" 提示条

## 六、非功能需求

- 安卓最低 API 26（Android 8.0）
- 本地数据库用 Room（或同类方案）
- 网络层自选（Retrofit、Ktor Client 均可）
- 后端语言/框架自选（Python FastAPI、Go、Node.js 均可）
- 数据库 MySQL（开发时可用 SQLite）
- 后端支持 Docker 部署
- 代码要有基本的类型安全（不要到处 any/dynamic）

1. **后端服务** — 可运行的 API 服务，含 Docker 部署配置
2. **安卓 APK** — 可安装运行的 debug 包

## 八、优先级

按这个顺序开发：

1. 后端 API + 数据库（先把接口跑通）
2. 安卓基础 UI + 网络调用（能连服务器用起来）
3. 离线功能（Room 本地存储 + 同步队列 + 冲突处理）
4. 提醒功能
5. 细节打磨（动画、错误处理、边界情况）

## 九、V2：Idea 好友圈

### 9.1 命名调整

- 客户端文案从 ToDo/任务逐步调整为 Idea
- 旧 API `/api/tasks` 保持兼容，新 API 增加 `/api/ideas`

### 9.2 好友圈

- 每个用户注册后自动生成唯一 `circleId`
- 用户只能拥有一个自己的好友圈 ID
- 用户可以通过别人的 `circleId` 加入对方好友圈
- 用户可以加入多个别人的好友圈
- 用户可以退出已加入的好友圈

### 9.3 Idea 可见性

- `private`：私密，仅自己可见
- `circle`：公开到自己的好友圈，加入自己好友圈的用户可以在 feed 里看到

### 9.4 Feed

- 新增好友圈 feed 页面
- 展示已加入好友圈中的公开 idea
- 支持按单个好友圈筛选
- feed item 显示作者、标题、内容、状态、分类、优先级、点赞数、评论数

### 9.5 点赞和评论

- 每个用户对同一个 idea 最多点赞一次
- 再次操作可取消点赞
- 支持一级评论
- 用户可以删除自己的评论
